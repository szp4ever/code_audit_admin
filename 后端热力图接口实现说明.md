# 用户操作热力图 - 后端实现说明

## 一、接口规范

### 1.1 接口路径
```
GET /api/taskmanagement/stats/user-operation-heatmap
```

### 1.2 请求参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| timeRange | string | 否 | 时间范围：`day`（最近1天）、`week`（最近7天）、`month`（最近30天），默认 `week` |
| startDate | string | 否 | 开始日期，格式：`YYYY-MM-DD`，与 `endDate` 配合使用 |
| endDate | string | 否 | 结束日期，格式：`YYYY-MM-DD`，与 `startDate` 配合使用 |

**参数优先级说明：**
- 如果提供了 `startDate` 和 `endDate`，则使用自定义时间范围
- 否则使用 `timeRange` 参数（默认 `week`）

### 1.3 响应格式

```json
[
  {
    "module": "system:user",
    "timeSlot": "2026-01-05",
    "count": 15
  },
  {
    "module": "taskmanagement",
    "timeSlot": "2026-01-05",
    "count": 8
  },
  {
    "module": "system:role",
    "timeSlot": "2026-01-06",
    "count": 12
  }
]
```

**响应字段说明：**
- `module`：功能模块标识，字符串类型
- `timeSlot`：时间段，格式为 `YYYY-MM-DD`（日期格式，不含时间部分）
- `count`：该模块在该时间段的使用频次（操作次数），整数类型

## 二、数据来源

### 2.1 数据表
从**操作日志表**（`sys_oper_log` 或类似表）中统计数据。

### 2.2 操作日志表字段映射

根据前端代码中的 `OperationLog` 接口定义，操作日志表应包含以下关键字段：

| 日志表字段 | 用途 | 说明 |
|-----------|------|------|
| `oper_url` | 提取模块标识 | 从URL路径中提取功能模块 |
| `oper_time` | 时间分组 | 用于按日期分组统计 |
| `status` | 过滤条件 | 建议只统计成功操作（status = 0 或 1） |

## 三、核心实现逻辑

### 3.1 功能模块标识提取规则

从操作日志的 `oper_url` 字段中提取功能模块标识，提取规则如下：

**URL路径到模块标识的映射规则：**

1. **系统管理模块**：
   - `/system/user` → `system:user`
   - `/system/role` → `system:role`
   - `/system/menu` → `system:menu`
   - `/system/dept` → `system:dept`
   - `/system/post` → `system:post`
   - `/system/dict` → `system:dict`
   - `/system/config` → `system:config`
   - `/system/notice` → `system:notice`
   - `/monitor/operlog` → `system:log:operation`
   - `/monitor/logininfor` → `system:log:login`

2. **系统监控模块**：
   - `/monitor/online` → `monitor:online`
   - `/monitor/cache` → `monitor:cache`
   - `/monitor/admin` → `monitor:admin`
   - `/monitor/job` → `monitor:job`

3. **工作流模块**：
   - `/workflow/category` → `workflow:category`
   - `/workflow/model` → `workflow:model`
   - `/workflow/define` → `workflow:define`
   - `/workflow/monitor` → `workflow:monitor`
   - `/workflow/form` → `workflow:form`

4. **任务相关模块**：
   - `/task/apply` → `task:apply`
   - `/task/todo` → `task:todo`
   - `/task/done` → `task:done`
   - `/task/cc` → `task:cc`
   - `/taskmanagement` → `taskmanagement`

5. **图谱管理模块**：
   - `/graph/instance` → `graph:instance`
   - `/graph/visualization` → `graph:visualization`
   - `/graph/rag` → `graph:rag`

6. **仪表盘模块**：
   - `/dashboard/analytics` → `dashboard:analytics`

**通用提取规则（建议）：**
- 提取URL路径的前两级作为模块标识
- 格式：`{第一级路径}:{第二级路径}`
- 如果只有一级路径，则使用：`{第一级路径}`
- 示例：
  - `/system/user/list` → `system:user`
  - `/monitor/operlog` → `monitor:operlog` → 映射为 `system:log:operation`
  - `/taskmanagement` → `taskmanagement`

### 3.2 时间范围计算

**当使用 `timeRange` 参数时：**
- `day`：统计最近1天的数据（从当前时间往前推1天）
- `week`：统计最近7天的数据（从当前时间往前推7天）
- `month`：统计最近30天的数据（从当前时间往前推30天）

**当使用 `startDate` 和 `endDate` 时：**
- 统计从 `startDate 00:00:00` 到 `endDate 23:59:59` 之间的数据

### 3.3 时间分组规则

- 按**日期**分组，格式为 `YYYY-MM-DD`
- 从 `oper_time` 字段中提取日期部分（去掉时间部分）
- 例如：`2026-01-05 14:30:00` → `2026-01-05`

### 3.4 统计逻辑

1. **过滤条件**：
   - 只统计操作成功的记录（`status = 0` 或根据实际业务定义）
   - 过滤掉删除操作（可选，根据业务需求）

2. **分组统计**：
   - 按 `模块标识` 和 `日期` 进行分组
   - 统计每个分组中的记录数量作为 `count`

3. **数据补全**（可选）：
   - 如果某个模块在某天没有操作记录，可以不返回该数据点
   - 前端会自动处理空数据

## 四、SQL 实现示例（MySQL）

```sql
-- 示例SQL（需要根据实际表结构调整）
SELECT 
    CONCAT(
        SUBSTRING_INDEX(SUBSTRING_INDEX(oper_url, '/', 2), '/', -1),
        ':',
        SUBSTRING_INDEX(SUBSTRING_INDEX(oper_url, '/', 3), '/', -1)
    ) AS module,
    DATE(oper_time) AS timeSlot,
    COUNT(*) AS count
FROM sys_oper_log
WHERE 
    status = 0  -- 只统计成功操作
    AND oper_time >= DATE_SUB(NOW(), INTERVAL 7 DAY)  -- 最近7天（根据timeRange动态调整）
    AND oper_time < DATE_ADD(CURDATE(), INTERVAL 1 DAY)
GROUP BY 
    module,
    timeSlot
ORDER BY 
    timeSlot ASC,
    module ASC;
```

**注意：** 上述SQL中的模块提取逻辑需要根据实际URL结构进行调整。

## 五、Java 实现建议

### 5.1 Controller 层

```java
@GetMapping("/stats/user-operation-heatmap")
@ApiOperation("获取用户操作热力图数据")
public R<List<UserOperationHeatmapItem>> getUserOperationHeatmap(
    @RequestParam(required = false) String timeRange,
    @RequestParam(required = false) @DateTimeFormat(pattern = "yyyy-MM-dd") LocalDate startDate,
    @RequestParam(required = false) @DateTimeFormat(pattern = "yyyy-MM-dd") LocalDate endDate
) {
    // 1. 计算时间范围
    LocalDate start, end;
    if (startDate != null && endDate != null) {
        start = startDate;
        end = endDate;
    } else {
        // 根据 timeRange 计算（默认 week）
        int days = "day".equals(timeRange) ? 1 : 
                   "month".equals(timeRange) ? 30 : 7;
        end = LocalDate.now();
        start = end.minusDays(days - 1);
    }
    
    // 2. 调用 Service 层查询
    List<UserOperationHeatmapItem> result = operLogService.getUserOperationHeatmap(start, end);
    
    return R.ok(result);
}
```

### 5.2 Service 层

```java
public List<UserOperationHeatmapItem> getUserOperationHeatmap(LocalDate startDate, LocalDate endDate) {
    // 1. 查询操作日志
    List<SysOperLog> logs = operLogMapper.selectList(
        new LambdaQueryWrapper<SysOperLog>()
            .eq(SysOperLog::getStatus, 0)  // 只统计成功操作
            .ge(SysOperLog::getOperTime, startDate.atStartOfDay())
            .le(SysOperLog::getOperTime, endDate.atTime(23, 59, 59))
    );
    
    // 2. 按模块和时间分组统计
    Map<String, Map<String, Long>> stats = logs.stream()
        .collect(Collectors.groupingBy(
            log -> extractModule(log.getOperUrl()),  // 提取模块标识
            Collectors.groupingBy(
                log -> log.getOperTime().toLocalDate().toString(),  // 提取日期
                Collectors.counting()  // 统计数量
            )
        ));
    
    // 3. 转换为返回格式
    List<UserOperationHeatmapItem> result = new ArrayList<>();
    stats.forEach((module, timeMap) -> {
        timeMap.forEach((timeSlot, count) -> {
            UserOperationHeatmapItem item = new UserOperationHeatmapItem();
            item.setModule(module);
            item.setTimeSlot(timeSlot);
            item.setCount(count.intValue());
            result.add(item);
        });
    });
    
    return result;
}

/**
 * 从URL中提取模块标识
 */
private String extractModule(String operUrl) {
    if (StringUtils.isBlank(operUrl)) {
        return "unknown";
    }
    
    // 移除查询参数
    String path = operUrl.split("\\?")[0];
    
    // 移除开头的斜杠
    if (path.startsWith("/")) {
        path = path.substring(1);
    }
    
    // 分割路径
    String[] parts = path.split("/");
    
    if (parts.length >= 2) {
        // 特殊映射处理
        String module = parts[0] + ":" + parts[1];
        
        // 特殊映射规则
        if ("monitor:operlog".equals(module)) {
            return "system:log:operation";
        }
        if ("monitor:logininfor".equals(module)) {
            return "system:log:login";
        }
        
        return module;
    } else if (parts.length == 1) {
        return parts[0];
    }
    
    return "unknown";
}
```

## 六、注意事项

1. **性能优化**：
   - 如果操作日志表数据量很大，建议在 `oper_url` 和 `oper_time` 字段上建立索引
   - 考虑使用定时任务预计算统计数据，避免实时查询

2. **数据准确性**：
   - 确保只统计有意义的操作（排除系统内部调用、健康检查等）
   - 根据实际业务需求决定是否排除某些操作类型

3. **模块标识一致性**：
   - 确保提取的模块标识与前端 `moduleMap` 中的键保持一致
   - 如果系统中有新的功能模块，需要同步更新提取规则

4. **时间处理**：
   - 注意时区问题，确保使用统一的时区
   - 日期格式必须为 `YYYY-MM-DD`，不含时间部分

5. **空数据处理**：
   - 如果查询结果为空，返回空数组 `[]` 即可
   - 前端会自动处理空数据并显示空图表
