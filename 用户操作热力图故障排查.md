# 用户操作热力图加载失败 - 故障排查指南

## 错误现象
前端显示：**"加载用户操作热力图数据失败"**

## 可能原因及解决方案

### 1. 后端接口未实现（最可能）

**症状：**
- 浏览器控制台显示 `404 Not Found` 或 `500 Internal Server Error`
- 网络请求失败

**检查方法：**
1. 打开浏览器开发者工具（F12）
2. 切换到 **Network（网络）** 标签
3. 刷新页面，查看请求 `/api/taskmanagement/stats/user-operation-heatmap`
4. 查看响应状态码：
   - `404`：接口不存在
   - `500`：服务器内部错误
   - `403`：权限不足

**解决方案：**
根据 `后端热力图接口实现说明.md` 文档实现后端接口。

**接口路径：**
```
GET /api/taskmanagement/stats/user-operation-heatmap
```

**请求参数：**
- `timeRange`（可选）：`day` | `week` | `month`，默认 `week`
- `startDate`（可选）：开始日期，格式 `YYYY-MM-DD`
- `endDate`（可选）：结束日期，格式 `YYYY-MM-DD`

**响应格式：**
```json
[
  {
    "module": "system:user",
    "timeSlot": "2026-01-05",
    "count": 15
  }
]
```

### 2. 接口路径不匹配

**症状：**
- 浏览器控制台显示 `404 Not Found`
- 请求路径与后端实际路径不一致

**检查方法：**
1. 查看前端 API 定义：`apps/web-antd/src/api/taskmanagement/index.ts`
2. 确认前端请求路径：`/taskmanagement/stats/user-operation-heatmap`
3. 检查后端 Controller 中的路径映射是否一致

**前端路径定义：**
```typescript
userOperationHeatmap = '/taskmanagement/stats/user-operation-heatmap'
```

**后端应该匹配：**
```java
@GetMapping("/stats/user-operation-heatmap")
```

**解决方案：**
确保前后端路径一致，注意：
- 前端会自动添加 `/api` 前缀（由 `baseURL` 配置）
- 后端 Controller 的 `@RequestMapping` 路径
- 最终完整路径应该是：`/api/taskmanagement/stats/user-operation-heatmap`

---

### 3. 响应数据格式不正确

**症状：**
- 请求返回 `200 OK`，但前端仍然报错
- 浏览器控制台可能显示数据解析错误

**检查方法：**
1. 查看网络请求的响应内容
2. 检查响应是否符合以下格式：

```json
[
  {
    "module": "string",      // 功能模块标识
    "timeSlot": "YYYY-MM-DD", // 日期格式
    "count": number          // 整数类型
  }
]
```

**常见问题：**
- 后端返回了包装对象，如 `{ code: 200, data: [...] }`，但前端期望直接返回数组
- 字段名不匹配（如 `module` vs `moduleName`）
- 数据类型错误（如 `count` 是字符串而不是数字）

**重要说明：**
根据前端请求拦截器的逻辑（`apps/web-antd/src/api/request.ts`），后端应该返回以下格式：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": [
    {
      "module": "system:user",
      "timeSlot": "2026-01-05",
      "count": 15
    }
  ]
}
```

拦截器会自动提取 `data` 字段返回给前端代码，所以前端实际接收到的是数组。

**解决方案：**
1. 如果后端使用了统一响应包装（如 `R<T>`），确保前端请求拦截器正确处理
2. 检查 `apps/web-antd/src/api/request.ts` 中的响应拦截器
3. 确保后端返回的数据结构与前端 `UserOperationHeatmapItem` 接口定义一致

**前端类型定义：**
```typescript
export interface UserOperationHeatmapItem {
  module: string;
  timeSlot: string;
  count: number;
}
```

### 4. CORS 跨域问题

**症状：**
- 浏览器控制台显示 `CORS policy` 相关错误
- 请求被浏览器阻止

**检查方法：**
查看浏览器控制台的错误信息，通常包含：
```
Access to XMLHttpRequest at '...' from origin '...' has been blocked by CORS policy
```

**解决方案：**
在后端配置 CORS，允许前端域名访问。

---

### 5. 认证/权限问题

**症状：**
- 浏览器控制台显示 `401 Unauthorized` 或 `403 Forbidden`
- 请求需要登录或特定权限

**检查方法：**
1. 确认用户已登录
2. 检查请求头中是否包含有效的 `Authorization` token
3. 确认后端接口是否需要特定权限

**解决方案：**
1. 确保用户已登录
2. 检查后端接口的权限配置
3. 如果接口需要特定角色或权限，确保当前用户具有相应权限

---

### 6. 请求参数格式问题

**症状：**
- 请求发送成功，但后端返回错误
- 后端日志显示参数解析失败

**检查方法：**
查看网络请求的 Query Parameters：
```
timeRange=week
```

**解决方案：**
确保后端能够正确解析查询参数：
- `timeRange` 应该是字符串类型
- 日期参数格式应该是 `YYYY-MM-DD`

---

2. 确认接口是否被调用
